Blocker #1 ‚Äî Docker-in-Docker

üê≥
üî¥ Hard Blocker ¬∑ Code Executor Service
Docker-in-Docker (DinD) Sandboxed Execution
Difficulty: Hard

The Code Executor Service needs to run user-generated code safely inside an isolated container. The naive approach ‚Äî spinning up Docker containers from within your Spring Boot service ‚Äî is called Docker-in-Docker and it will immediately hit a wall on most developer machines.

Your 4-Stage Learning Journey

1
BREAK IT ‚Äî Build the naive approach first
Mount the Docker socket into your Spring Boot container and use the Docker Java SDK to spin up child containers. Run it. You will get a permission denied error on the Docker socket, or the container will start but have no network access, or it will escape resource limits silently.
# What you'll write first (naive) # docker-compose.yml code-executor: volumes: - /var/run/docker.sock:/var/run/docker.sock # This is the problem
What breaks: The Docker daemon runs as root. Your service doesn't. The socket mount works on Linux but behaves differently on Docker Desktop (Mac/Windows). And even when it works ‚Äî any code run inside can see the host Docker socket and escape the sandbox completely. This is a real CVE-class vulnerability.
2
UNDERSTAND IT ‚Äî Why DinD is genuinely hard
Spend time understanding the Docker architecture ‚Äî the daemon, client, socket, and namespace isolation. Learn what --privileged mode actually does (it disables ALL Linux security modules ‚Äî seccomp, AppArmor, capabilities). Understand why companies like Google Cloud Run, AWS Lambda, and Replit use separate VM-level isolation (gVisor, Firecracker) not DinD for user code execution. This is exactly the kind of infrastructure knowledge that separates mid-level from senior engineers.
3
FIX IT ‚Äî Proper DinD with rootless Docker + resource limits
Implement it correctly using rootless Docker mode + the Docker Java SDK with explicit resource constraints. This is what you'll learn from the Java Executor Framework + Docker modules in the cohort.
// Code Executor Service ‚Äî proper implementation @Service public class DockerExecutorService { private final DockerClient dockerClient; public ExecutionResult execute(String code, String language) { // 1. Create container with hard resource limits CreateContainerResponse container = dockerClient .createContainerCmd("code-runner:" + language) .withHostConfig(HostConfig.newHostConfig() .withMemory(128 * 1024 * 1024L) // 128MB max RAM .withCpuQuota(50000L) // 50% of one CPU .withNetworkMode("none") // NO network access .withReadonlyRootfs(true) // read-only filesystem .withCapDrop(Capability.ALL)) // drop all Linux caps .exec(); // 2. Run with Java Executor + hard timeout Future<String> future = executor.submit(() -> runContainer(container)); return future.get(10, TimeUnit.SECONDS); // 10s timeout } }
4
PRODUCTION THINKING ‚Äî How Replit/Lovable actually do it
Research and document how real platforms solve this at scale. Replit uses gVisor (Google's user-space kernel). AWS Lambda uses Firecracker microVMs. Understand why your DinD implementation is a valid learning exercise but would need to evolve for real production traffic. This becomes a talking point in interviews where you can say "I understand the tradeoffs between DinD and VM-level isolation and why companies choose one over the other."

"I built a sandboxed code execution service for an AI SaaS platform. My first approach was Docker socket mounting ‚Äî I hit privilege escalation issues and understood why that's a CVE risk. I then implemented rootless Docker with capability dropping, memory limits, and no-network mode using the Docker Java SDK. I also researched how production systems like Replit and AWS Lambda use gVisor and Firecracker for stronger isolation."

Blocker #2 ‚Äî JWT Across Microservices

üîê
üî¥ Hard Blocker ¬∑ Auth + All Services
Inter-Service JWT Propagation & Trust Model
Difficulty: Hard

You've planned JWT for the API Gateway ‚Äî great. But the moment one microservice needs to call another (AI Gen Service calls Project Service to save generated code), you hit a design gap: who validates the token? Does Service B trust Service A blindly? What if someone calls Service B directly bypassing the Gateway?

1
BREAK IT ‚Äî Build with no inter-service auth first
Implement the Gateway JWT validation. Then have AI Gen Service call Project Service via OpenFeign with no token. It works fine. Now directly hit Project Service from Postman (bypassing Gateway) ‚Äî it will happily accept the request. Your security is gateway-only and you have no defense-in-depth. This is the exact security mistake that caused real data breaches in microservice architectures.
2
UNDERSTAND IT ‚Äî The 3 patterns every team debates
Study the three real approaches to inter-service security and understand the tradeoffs of each:

Pattern A ‚Äî Gateway-only trust (network perimeter): Only the Gateway validates tokens. Internal services trust all internal network calls. Simple. Works only if your internal network is truly private. Breaks in cloud environments where services are publicly reachable.

Pattern B ‚Äî JWT forwarding (what you'll implement): Gateway validates and forwards the original JWT header. Each service also validates JWT on every call. Defense-in-depth. Zero-trust minded. More CPU (verify on every hop) but genuinely secure.

Pattern C ‚Äî Service-to-service tokens (mTLS / service accounts): Each service has its own identity token separate from the user token. This is what Kubernetes service accounts and Istio service mesh do. Production-grade but complex to set up.
3
FIX IT ‚Äî Pattern B with OpenFeign RequestInterceptor
Implement JWT forwarding using OpenFeign's RequestInterceptor. This automatically propagates the user's JWT to every downstream service call ‚Äî and each service validates it independently.
// 1. OpenFeign interceptor ‚Äî propagates JWT to downstream calls @Component public class FeignJwtInterceptor implements RequestInterceptor { @Override public void apply(RequestTemplate template) { ServletRequestAttributes attrs = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes(); if (attrs != null) { String jwt = attrs.getRequest().getHeader("Authorization"); if (jwt != null) { template.header("Authorization", jwt); // forward token } } } } // 2. Each service validates JWT independently @Component public class JwtAuthFilter extends OncePerRequestFilter { @Override protected void doFilterInternal(...) { String token = extractToken(request); if (jwtService.isValid(token)) { // set SecurityContext ‚Äî service is now auth-aware SecurityContextHolder.getContext() .setAuthentication(createAuth(token)); } filterChain.doFilter(request, response); } }
4
PRODUCTION THINKING ‚Äî Zero Trust & mTLS
Research Istio service mesh and how it implements mutual TLS (mTLS) between services automatically ‚Äî no JWT forwarding needed because the network itself is authenticated. Understand how Kubernetes RBAC + service accounts works. Document the evolution: your Pattern B ‚Üí production Pattern C. This is senior-level architecture thinking.
üí¨ What You Can Say in an Interview
"How did you secure communication between your microservices?"
"I first built it with gateway-only JWT validation and demonstrated the vulnerability ‚Äî you could bypass the gateway and call internal services directly. I then implemented JWT forwarding via OpenFeign's RequestInterceptor with independent validation on each service. I also researched how production systems use mTLS with Istio service mesh as the next evolution, which eliminates the need for application-level token propagation entirely."

pgvector ‚Äî Not in Default PostgreSQL Image
Difficulty: Medium

pgvector is a PostgreSQL extension. It doesn't ship with the standard postgres:latest Docker image. If you pull standard postgres, CREATE EXTENSION vector; will fail with "extension not found." You will waste time thinking it's a Spring configuration problem when it's actually an image problem.

Break ‚Üí Understand ‚Üí Fix

Break it: Use postgres:16 and try to run CREATE EXTENSION vector;. It fails. Now you understand that Docker images are not monolithic ‚Äî extensions must be explicitly included or compiled in.

Understand it: Learn how PostgreSQL extensions work ‚Äî they're C libraries loaded into the server process. shared_preload_libraries, pg_extension catalog, how extensions add functions and types to PostgreSQL's type system.

Fix it: Switch to ankane/pgvector which pre-compiles the extension. Then manually install it in a standard image so you understand exactly what that Docker image is hiding from you.

# docker-compose.yml ‚Äî correct image postgres-vector: image: ankane/pgvector:latest environment: POSTGRES_DB: vibecode_ai volumes: - ./init.sql:/docker-entrypoint-initdb.d/init.sql -- init.sql ‚Äî run once on startup CREATE EXTENSION IF NOT EXISTS vector;

Ollama ‚Äî Model Size vs Your VRAM Budget
Difficulty: Medium

With 4-6GB VRAM you need to be deliberate about model selection. Pulling the wrong model means it falls back to CPU inference ‚Äî which is 10-20x slower and will make your app feel broken during development.

Break ‚Üí Understand ‚Üí Fix

Break it: Pull ollama pull llama3.1:70b and watch it either refuse to load on GPU or crawl at 0.5 tokens/sec on CPU. Then understand why ‚Äî quantization levels, VRAM math.

Understand it: Learn model quantization ‚Äî Q4, Q5, Q8 mean 4-bit, 5-bit, 8-bit weights. A 7B parameter model at Q4 = ~4GB, at Q8 = ~7GB. VRAM must hold the full model for GPU inference. Learn how CPU offloading works (partial GPU, partial RAM).

Fix it ‚Äî Your optimal model stack:

# Primary ‚Äî Code generation (fits your VRAM perfectly) ollama pull codellama:7b-instruct-q4_K_M # ~4GB VRAM, fast # Secondary ‚Äî General chat/RAG (smaller, faster) ollama pull mistral:7b-instruct-q4_K_M # ~4GB VRAM # Spring AI config ‚Äî swap OpenAI for Ollama (1 line change) # application.yml spring: ai: ollama: base-url: http://localhost:11434 chat: options: model: codellama:7b-instruct-q4_K_M embedding: options: model: nomic-embed-text # for RAG embeddings

8 Services Running Locally ‚Äî Memory Explosion
Difficulty: Easy to Miss

8 Spring Boot services + Kafka + Zookeeper + PostgreSQL + MongoDB + Redis + Ollama running simultaneously on 16GB RAM will leave you ~2GB free. Your machine will start swapping to disk and everything will feel broken. This is a real operational problem even senior engineers hit.

Break ‚Üí Understand ‚Üí Fix

Break it: Spin everything up at once via Docker Compose. Watch your RAM hit 90%, your IDE slow to a crawl, Kafka start dropping messages. Understand that this is a real production concern ‚Äî not just a dev machine problem.

Fix it: Set explicit JVM heap limits per service in Docker Compose. Spring Boot doesn't cap JVM heap by default ‚Äî it'll grab as much as it can.

# docker-compose.yml ‚Äî JVM limits per service auth-service: environment: JAVA_OPTS: "-Xms128m -Xmx256m" # 256MB max heap ai-generation-service: environment: JAVA_OPTS: "-Xms256m -Xmx512m" # needs more for AI work # Kafka + Zookeeper memory kafka: environment: KAFKA_HEAP_OPTS: "-Xmx256m -Xms128m"

Also: start services in groups during development ‚Äî you don't need Billing + Notification running while working on Auth. Use Docker Compose profiles to group services.

Service Registry (Eureka) ‚Äî Correct Pattern
All services registering with Eureka + Gateway using service names (not hardcoded URLs) is the right pattern. When you move to Kubernetes, Eureka gets replaced by K8s DNS ‚Äî same concept, different implementation. You'll understand why when you get there.

Database-per-Service ‚Äî Correct Microservice Principle
Auth/User/Billing on PostgreSQL, Projects on MongoDB, Vectors on pgvector ‚Äî each service owns its data. No shared database tables across services. This enforces loose coupling. You'll feel the benefit when you change the Project schema and zero other services break.
